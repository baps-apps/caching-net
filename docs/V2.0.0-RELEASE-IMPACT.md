# Caching.NET v2.0.0 Release Impact

This document summarizes what changed in v2.0.0 and why it matters from both business and technical perspectives.

## Executive Summary (Business)

Caching.NET v2.0.0 upgrades the platform from a basic shared cache helper to a production-grade cache runtime designed for high-throughput services. The release improves reliability under failure, observability in production, security defaults, and operational readiness for modern deployment environments.

### Business Outcomes

- Faster and more stable user-facing APIs through better cache hit paths and reduced stampede risk.
- Lower incident recovery time via standardized telemetry and richer diagnostics.
- Stronger security posture out-of-the-box (strict TLS, redaction, SBOM).
- Better release confidence through expanded automated test coverage and local CI-equivalent scripts.
- Clearer adoption path for teams moving from v1 (`docs/MIGRATION-V1-TO-V2.md`).

## What Changed (Technical)

## Core Runtime and API

- Unified `ICacheService` contract across `InMemory`, `Redis`, and `Hybrid`.
- Expanded async API surface:
  - `GetAsync<T>()`, `ExistsAsync()`, `RefreshAsync<T>()`
  - `GetManyAsync<T>()`, `SetManyAsync<T>()`, `RemoveManyAsync()`
- Per-call behavior controls through `CacheCallOptions` (expiration, stale window, jitter, tags).
- Canonical key construction via `CacheKey.For<T>(id)...Build()`.
- Fluent presets: `Enable()`, `UseDevelopmentDefaults()`, `UseProductionDefaults()`, `WithKeyValidator`, `WithKeyTransformer`; startup validation enforces **prefix + user-key budget** for `MaximumKeyLength`.

## Reliability and Resilience

- Striped lock coalescing to prevent stampede amplification (xxHash32-over-UTF8 stable stripe selection).
- Polly resilience pipelines (optional concurrency limiter, timeout, circuit-breaker, retry) for Redis operations; widened transient detection (socket I/O, Redis failover messages); tighter retry backoff (50 ms → 1 s with jitter).
- Stale-While-Revalidate orchestration with bounded background refresh; **`StaleEntryTracker`** prunes on a registration cadence and caps dictionary size.
- **`RoutingCacheService`** implements **`IAsyncDisposable`** and awaits in-flight background stale refreshes on shutdown.
- Redis connection rotation support via `RedisConnectionRotator` on options change (drain-delayed multiplexer teardown).
- **Built-in health checks** (internal implementations; register via `WithHealthChecks` / `AddCachingHealthChecks` only): Redis/Hybrid readiness path requires a connected multiplexer, **`PING`** before the cache probe, and a **per-process probe key** (`MachineName` + `ProcessId`).
- **`Enabled=false`**: no backends or Polly registry registered; validator skipped; `ICacheService` still resolves and short-circuits. After a cold start with caching off, **restart** when turning caching on so backends register (see [MIGRATION-V1-TO-V2.md](MIGRATION-V1-TO-V2.md)).

## Observability

- OpenTelemetry-native instrumentation through `CacheInstruments` (subscribe to `Meter("Caching.NET")`; `ActivitySource` is public but **no spans are started** in v2).
- Dedicated **serializer latency** histograms: `cache.serialize.duration`, `cache.deserialize.duration` (tagged by format).
- Envelope and schema drift visibility for Redis payload compatibility issues; drift **logs** are rate-limited by **`DriftLogSampler`** (per drift kind + key fingerprint, one minute).
- TLS validation outcome telemetry (`cache.tls.validation`).
- Source-generated logging for lower overhead on hot paths.
- `RemoveManyAsync` on Redis (multiplexer path) records **`cache.removes` by actual deleted key count**.

## Performance and Scale

- **`ICacheSerializer.Deserialize<T>(ReadOnlyMemory<byte>)`** enables Redis MessagePack reads **without** an extra `ToArray()` copy when decoding envelope payloads; JSON uses `bytes.Span` into the same backing array.
- TTL jitter support to reduce synchronized expiry storms.
- Server-side Redis batch optimization path with multiplexer integration.
- In-memory **batch APIs** use synchronous `IMemoryCache` loops where appropriate (no per-key `await` overhead).
- **`PayloadEnvelope.Write`** supports **`IBufferWriter<byte>`** to avoid allocating a full wire `byte[]` on hot encode paths; decode enforces **exact payload length** vs. declared `PayloadLen`.
- Benchmark suite + perf gate scripts for regression control.
- AOT/trim smoke project to validate publish-time compatibility paths.

## Security and Compliance

- Strict Redis certificate validation enabled by default.
- Connection-string redaction in validation/error surfaces.
- **`CacheOptionsValidator`** (when enabled) parses Redis configuration via **`ConfigurationOptions.Parse`** and requires at least one endpoint; enforces **prefix + `MaximumKeyLength` budget** so user segments retain minimum space.
- High-cardinality telemetry tag protections via analyzer rules (`CN0001`), including logger templates.
- SPDX 2.2 SBOM emitted in package artifacts.

## Builder and validation (post-audit)

- **`CachingBuilder`**: `Enable()`, `UseDevelopmentDefaults()` / `UseProductionDefaults()`, `WithKeyValidator` / `WithKeyTransformer`; configured via `AddCaching(...)`.

## Quality and Operations

- New test layers:
  - Integration tests (real Redis via Testcontainers)
  - Chaos tests (fault and breaker behavior)
  - Property tests (FsCheck invariants)
- Local CI-equivalent workflow via `scripts/dev.ps1` (`build`, `test`, `test:integration`, `test:chaos`, `test:property`, `aot`, `bench`, `bench:gate`, `pack`, `all`).
- Documentation set rewritten for v2 operations, internals, telemetry, security, and migration.

## Improvement Matrix

| Area | v1.x | v2.0.0 Improvement |
|---|---|---|
| Performance | Basic caching behavior | Batch APIs, jitter, SWR orchestration, benchmark gating |
| Reliability | Limited fault handling | Polly pipelines, coalescing, fail-open controls, rotation hooks |
| Observability | Narrow telemetry model | OTel-native metrics/logging and drift/TLS instrumentation |
| Security | Less strict defaults | Strict TLS default, redaction, analyzer safeguards, SBOM |
| Operability | Basic docs/workflow | Full runbooks + local pre-tag gate scripts |
| Test Confidence | Unit-heavy | Integration + chaos + property + AOT validation |

## Expected Production Impact

- Reduced p95/p99 latency variability in high-concurrency paths.
- Fewer cache-related cascading failures during backend turbulence.
- Better deployment safety during schema/serializer evolution.
- Faster troubleshooting due to standardized telemetry and deterministic scripts.
- Improved audit/compliance readiness through explicit supply-chain artifacts.

## Adoption Notes

- v2 is a major release with breaking changes; use `docs/MIGRATION-V1-TO-V2.md`.
- Set `Caching:KeyPrefix` explicitly in all environments (mandatory).
- For `Redis`/`Hybrid`, provide `Caching:RedisConnectionString`.
- Wire OpenTelemetry in host startup to collect v2 instrumentation.
